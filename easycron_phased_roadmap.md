# ⏳ easycron: NPM Package CLI Roadmap & Edge Case Analysis

This document outlines the phased delivery plan for the `easycron` NPM CLI, thoroughly detailing the Core UX, User Stories, API agreements, and Critical Edge Cases.

---

## 🎯 Core Positioning
**easycron** is explicitly built to solve the free-tier server-sleep problem using a two-tier approach. They are not equal priorities:
1. **Primary Mode:** External Trigger *(Reliable Scheduling for free-tier deployments)*
2. **Secondary Mode:** Local Cron *(Convenience, local testing, and always-on servers)*

> ⚠️ **Why not just use internal cron?** 
> *Internal cron tools (like `node-cron`) fundamentally fail on free-tier platforms like Render or Railway because the platform forcefully puts the server to "sleep" after 15 minutes of inactivity. When the server sleeps, the background cron dies with it. The `easycron external` mode solves this by triggering your sleeping API from an awake outside source.*

### 🚫 Non-Goals
To maintain project clarity and scope, **easycron** is explicitly:
- **Not a job queue system:** It does not handle job execution concurrency, queues, or distributed workloads (e.g., BullMQ/Celery).
- **Not a server uptime guarantee tool:** It triggers routes externally, but cannot magically prevent platforms from hard-terminating containers at the end of billing cycles.
- **Not a replacement for enterprise schedulers:** It is a webhook trigger tool, not an orchestration daemon like Apache Airflow.

### 🛑 Limitations
- **Dependent on External Providers:** Tightly coupled to GitHub Actions or UptimeRobot architecture; if their cron services lag, `easycron` lags.
- **Not Real-Time Guaranteed:** Action schedules can be delayed natively by GitHub during peak hours.
- **Subject to Network Reality:** Vulnerable to external DNS failures, sudden free-tier routing drops, or SSL handshake timeouts entirely outside the CLI's control.

### 🧠 Decision Tree
* **If server is always ON** → Use local cron (`easycron "schedule" -- command`)
* **If server sleeps (Free Tier)** → Use external trigger (`easycron external <url> "schedule"`)
* **If task is hyper-critical** → Use redundancy mode (`--redundant` multi-trigger)

## ⭐ Golden Path (The "Happy Path" User Journey)

### 0. Quick Installation
```bash
npm i -g easycron
```

The defining workflow that developers will experience within 60 seconds of installation:

1. **User runs command:**
   `easycron external https://api.my-app.com/task "every 10 minutes"`
2. **CLI generates logic:**
   CLI parses English translation and generates `.github/workflows/easycron-trigger.yml`.
3. **User commits file:**
   User pushes the file to their GitHub repository.
4. **GitHub external trigger initiates:**
   GitHub securely acts as the cron host, triggering a webhook every 10 minutes.
5. **Server wakes & task runs:**
   The sleeping server starts up, receives the hit, and executes the business logic.

## 💻 Command & Input UX

A centralized view of the developer tooling:

- `easycron external <url> "<schedule>"` *(Generates GH/Web triggers)*
- `easycron "<schedule>" -- <command>` *(Schedules a local cron job)*
- `easycron list` *(Shows active local jobs)*
- `easycron remove <id>` *(Removes a local job)*
- `easycron explain "<schedule>"` *(Dry-rehearses the cron parsing translation)*

### 📝 Supported NLP Schedule Inputs:
The CLI uses deterministic parsing to convert the following plain English into standard cron formats:
- `every 10 minutes`
- `every 2 hours`
- `daily at 2am` *(or `daily at 02:00`)*
- `every monday at 9am`
- `every weekday at 14:00`

### 🎨 CLI Output Example (UX Feedback)
When the user runs the `external` command, the terminal visual feedback must be pristine and informative:
```console
✔ Schedule parsed: every 10 minutes
✔ Cron: */10 * * * *
✔ GitHub Action generated at: 
  .github/workflows/easycron-trigger.yml

⚠️ Warning: Running <5 min intervals may exceed GitHub free tier limits
```

### 📄 Example GitHub YAML Output
This is the baseline structure generated by the CLI, keeping developers visually grounded in what happens behind the scenes:
```yaml
name: easycron-trigger
on:
  schedule:
    - cron: "*/10 * * * *"

jobs:
  trigger:
    runs-on: ubuntu-latest
    steps:
      - name: Hit Endpoint with Smart Retry
        run: |
          for i in {1..3}; do
            curl -m 10 -f https://api.my-app.com/task && break
            echo "Request failed, retrying in 10s..."
            sleep 10
          done
```

## 🔄 API Contract, Security & Reliability

### 1. API Contract (Integration Expectation)
When using the "External Trigger" flow, the target API endpoint MUST adhere to this contract:
- **Endpoint:** `GET` or `POST` `/api/task`
- **Response:** Must respond instantly with `200 OK`
- **Behavior:** Respond immediately to the network request to prevent GitHub/Uptime thresholds from timing out, then execute the heavy task **asynchronously** in the background.

### 2. Time Zone Behavior (Critical Edge Case)
This is vital to document clearly so users don't miscalculate triggers:
- **Local execution (`easycron ... -- command`)** uses the **system's local timezone** by default.
- **External execution (`easycron external`)** utilizing GitHub Actions strictly operates on **UTC time**. 
- Developers must manually offset their required times for external flows, and the CLI `explain` command should actively highlight that GitHub triggers fire in UTC.

### 3. Security Consideration
If anyone finds the webhook URL, they can hit it aggressively. To secure the endpoint, `easycron external` must support basic token templating:
- **Query Parameter Approach:** `curl https://api.my-app.com/task?key=SUPER_SECRET`
- **Header Token Support:** Passing an `--auth="Bearer TOKEN"` CLI flag configures secure headers directly in the exported GH Action YAML.

### 4. Idempotency & Rate Limiting Strategies (App Side)
Because external triggers utilize retry loops and redundancy mode, multiple duplicate hits *will* land on the application server in rapid succession.
- **Requirement:** Developers must actively avoid processing duplicate executions. 
- **Implementation Strategy:** App logic must utilize in-memory execution locks, database boolean flags (e.g., `is_running = true`), or Redis/Cache cooldown windows (e.g., 1-minute debounces). This code isn't built into easycron, but must be documented for developers implementing easycron endpoints.

### 5. Failure Handling Flow
When everything fails, developers must have a clean trail to follow:
1. **GitHub Action Executes** (Target sleepy app doesn't respond).
2. **Retry Logic Triggered** (`curl` bumps 3x over 10-30s intervals).
3. **If still fails:**
   - 🔴 Action exits with status code 1.
   - 🔴 Logs are permanently visible in the GitHub Actions UI tab for diagnosis.
   - 🔴 User checks their native Server logs (Render/Railway) using the timestamp provided in GitHub Actions to trace back to an infrastructural freeze or network crash.

## 🟢 Phase 1: Core CLI & MVP Implementation (Weeks 1–2)
**Goal:** A fast, functional CLI tool for local scheduling and pure cron-parsing, proving viability and establishing the `@easycron/cli` NPM scope.

### Epic 1.1: Human-Friendly Parsing Engine
- **Must Have:** Support basic intervals ("every 10 minutes", "every 2 hours", "daily at HH:MM").
- **Must Have:** Standard CLI error handling that parses input arrays cleanly over raw regex fails.
- **Edge Cases:** Regex trailing spaces (`every    10   minutes`), case insensitivity, rejecting floating-point floats.

### Epic 1.2: Local Job Execution & Persistence (Lean MVP)
- **Must Have:** Execute commands natively via `child_process.exec` using `node-cron`.
- **Must Have:** Basic JSON storage. Creating and reading from `~/.easycron/jobs.json`.
- **Edge Cases:** Foreground terminal persistence (explicitly warning users `easycron` local mode dies if terminal closes). 
- **Deferred (Later):** File locking, concurrency safety across multiple terminals, and advanced corruption recovery mechanisms are deferred from MVP to avoid over-engineering the launch.

---

## 🟡 Phase 2: External Triggers for Free Tier (Weeks 3–5)
**Goal:** Deliver the primary value proposition—generating trigger logic for external webhooks.

### Epic 2.1: External Codegen Generation
- **Must Have:** Codegen for `.github/workflows/easycron-trigger.yml` via the `easycron external` command.
- **Must Have:** Output raw configs for UptimeRobot or cron-job.org as fallback tiers.
- **Redundancy Mode Implementation:** Running `easycron external <url> "every 10 min" --redundant` generates both a GitHub Action and an UptimeRobot config locally, giving the developer layered HTTP triggers in case GitHub experiences downtime.
- **Edge Cases to Handle:** URL validation (silently rejecting `localhost`). Preventing dangerous `.yml` overwrite collisions.
- ⚠️ **CRITICAL WARNING:** CLI output logging must fiercely warn against aggressive polling intervals.

### Epic 2.2: Advanced Infrastructure Resilience
- **Must Have:** A built-in Bash 3x retry hook inside the GH Action configuration to compensate for Render/Railway "sleeping cold starts".
- **Must Have:** Script must differentiate `curl` HTTP network timeouts (to utilize retries) vs 5xx application crashes (to fail immediately). 

---

## 🟣 Phase 3: Developer Polish & Advanced Scaffolding (Weeks 6+)
**Goal:** Scale the package through automation scaffolding and advanced architecture.

### Epic 3.1: DX Code Scaffolding
- **Must Have:** Scaffold tools teaching the "Fast Endpoint" convention. Generating an Express.js boilerplate endpoint showing async detachments to honor the API Contract.
- **Later Roadmap:** Advanced state recoverability (the `jobs.json` file locking/corruption recovery deferred from Phase 1).

### Epic 3.2: Open Source Extensibility 
- **Must Have:** Move parsing definitions to a configurable plugin architecture. 
- **Edge Cases:** Security profiling to avoid Regex Denial of Service (ReDoS) from faulty community submissions and sandboxing community AST definitions.
